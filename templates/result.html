Checklist:

1. Financially stable
 Allow for a well-financed cloud provider that is always having money fed back into it to prepare for any states of monetary hiccups the business may fall into over time.

  • If the business is to go under, there may not be the finance to support accessing the cloud services via the business
  • This could lead to further migration to a new cloud provider that can be time consuming and costly


2. Research into the provider
 It is always worth, before committing to a long-term relationship with someone, to do your research.

  • When looking for the ideal cloud service provider a lot of things apply. For one, their current customers should be happy and able to recommend it. Look online for reviews from existing customers
  • The provider you’re thinking of should be popular among its competitors in the market


3. The location of datacentres for the cloud provider
 For increased security of your cloud-based applications cloud providers allow for the option to allow geographical distribution of your data

  • The most secure and widely used servers are in Asia, Europe and America
  • Having servers in multiple regions means if a region goes down there are others to rely on for 24/7 access to cloud applications

4. Standards of Cloud Security
 The exchanging of any data via the internet is to be end-to-end encrypted, this means the packets of data you are sending cannot be read in plain text by a potential hacker. This is imperative when dealing with sensitive customer data as a business.

  • - The provider should always be encrypting data during the transfer to the cloud and within the server itself
  • - When the data is encrypted end-to-end the developer cannot even view a files content
It is worth checking the cloud provider you are interested in has been certified in its practise of safely storing the data of its clients (Your data)
  • SSAE is the most common online auditing standard available for companies to have
  • By a cloud provider having an auditing standard, this verifies that the service has been certified for its safe practices of storing data

It was found that any additional on-premises applications (Integration points) connected to the system that’s being moved to the cloud can make the transition to the cloud more complex. The study found that one participant had analysed a banking application from within their business to be potentially moved to the cloud but found over 50 integration points and resulted in it not being migrated to the cloud. These integration points are applications connected to the cloud structure that still run legacy applications. Although this is possible and you can have legacy systems integrated with a cloud structure it can cause latency and lag when both systems try to communicate with each other, reducing overall user experience. When an application that’s deployed to the cloud from a legacy system is still communicating with applications not in the same environment, it can lead to network latency issues.

Applications with fewer integrated systems attached to them have a much better chance of being able to migrate to the cloud. The more integration points in an application with other applications, the greater the latency between the two systems.

It is important that during analysis of undergoing a legacy application migration, the number of applications the system interacts with should be recorded. As well as the location of these integrated applications, the frequency of communications between any integrated systems & the amount of bandwidth used when transferring data between the systems.

The level of change required for recreating a legacy application in a cloud structure may determine if its eligible to be transferred to the cloud, if there appears to be no additional value is included in the work. From a business perspective, if the cost of reworking legacy systems to fit a cloud structure is additional with no apparent benefit it may be not be worth carrying out the transfer. The system would have the same capability as before being moved to the cloud, only technically working different. The best applications to be moved to the cloud are the applications that don’t require too much effort in repurposing to migrate to the cloud.

In the case that migrating your systems to the cloud requires rework and additional cost. It is worth working out how much this potential investment could benefit the business.

It is typical for businesses to go over an arranged budget for migration to the cloud when a plan is first formulated. The business moving their applications to a SaaS (Software as a Service) model cloud structure is likely to see costs go up. However, if there are any extra costs to running a cloud structure for the business applications it can be weighed up using a ROI (Return on Investment) calculator or the steps provided in this website.

It is important that the overall costs are worked out using the previous steps of both the businesses before migrating and how much the cloud will cost the business in the long term. This study has shown that from experience, the initial costs of entry level cloud operations is low. However, inevitably operating a cloud application for a business to naturally grow will over the course of time will have higher requirements and specifications which can end up costing as much, if not more to run than the old systems.

Although this seems to be the more-than likely fate of moving to the cloud, its not without its benefits. These costs tie in with the business’s newfound capabilities and facilities because of what working a business from the cloud provides.

Checklist for calculating ROI (Return on investment)

Step 1: Monitor requirements

  • Have access to the costs of different cloud providers you are interested in, base this on the specific parts of your system you want on the cloud
  • Monitor the businesses performance, capacity and availability and save it in a profile
  • It could be worth using the businesses last year of history for these areas to collect a comprehensive profile
  • Create a list of different security and performance requirements for your business based off these detailed profiles

Step 2:

  • Have access to your businesses expenses and payment records for evaluating overall prices for different cloud vendors
  • Pricing varies between cloud vendors as they all offer different pricing models
  • Look for vendors that adhere to the business’s capability needs (Performance & Security)
  • Price can depend on the CPU or Network Utilisation of the legacy application
  • Be aware and keep an eye out for volume discounts offered
  • If it appears that your business has strict performance requirements that cannot be supported by the cloud vendors you are interested in, it could mean that a dedicated environment may be necessary
  • This can drive up the cost of migration by a lot as dedicated systems and processes have to be developed for the specific system to work with cloud operations

Step 3: Total cost of ownership

  • Utilise the easy-to-use, total cost of ownership calculators.
  • Most of the top providers allow users to create their own cost model based on the profile that’s been created with the information about their business
  • These models allow for specific details about a business, figuring things out such: as how much storage is required & how much bandwidth is predicted to be used

Step 4:

  • Now that both the costs of moving to the cloud and staying on-premises applications have been explored, the data collected should be in a format that is suitable to be analysed in a spreadsheet
  • The spreadsheet is to be set up to weigh out the costs of every scenario, this will enable a return figure to be calculated
